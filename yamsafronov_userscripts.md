## [ Сценарий 1 – Авторизация и регистрация пользователя ]

1. Система отображает стартовый экран с выбором: "Вход" или "Регистрация".
2. При регистрации пользователь вводит Имя, Email и Пароль.
3. Система на стороне клиента проверяет формат почты и длину пароля.
4. Пользователь подтверждает ввод, и данные отправляются на сервер.
5. Сервер проверяет, не занят ли Email другим пользователем в базе данных.
6. При успешной регистрации система создаёт новую запись пользователя.
7. При успешной авторизации или регистрации система создаёт сессию пользователя.
8. Система выполняет переход:
   - к сценарию 2 (при регистрации)
   - на главный экран (при входе)



## [ Сценарий 2 – Настройка профиля аллергенов ]

1. Пользователь нажимает на иконку "Профиль" в нижней навигационной панели приложения или попадает на этот экран при регистрации.
2. Пользователь видит интерактивный список категорий ("Орехи", "Глютен" и т. д.).
3. Пользователь выбирает нужные категории, нажимая на соответствующие элементы.
4. Система помечает выбранные элементы при помощи закрашивания ячеек.
5. Пользователь нажимает кнопку "Готово".
6. Система отправляет массив выбранных данных на сервер, и он обновляет настройки пользователя в базе данных.
7. Система перенаправляет пользователя:
   - на главный экран (если это первичная настройка)
   - обратно в профиль (если редактирование)



## [ Сценарий 3 – Сканирование штрих-кода ]

1. Пользователь нажимает кнопку "Сканировать товар" на главном экране.
2. Система запрашивает разрешение на доступ к камере (при первом запуске).
3. Пользователь наводит объектив устройства на штрих-код продукта.
4. Штрих-код считывается и передается на сервер для анализа.
5. Сервер находит продукт и сверяет его состав с личным списком аллергенов пользователя.
6. Система отображает карточку с подробной информацией о товаре, если он найден в базе, а иначе запускается Сценарий 4.
7. В верхней части карточки выводится однозначный вердикт: "Безопасно" или "Опасно".



## [ Сценарий 4 – Добавление отсутствующего товара ]

1. Пользователь сканирует товар, информация о котором отсутствует на сервере.
2. Система отображает экран с уведомлением: "Продукт не найден в базе".
3. Пользователь нажимает кнопку "Добавить информацию о товаре".
4. Система открывает форму с предзаполненным номером штрих-кода.
5. Пользователь вводит название товара и название бренда.
6. Пользователь вводит текст состава с упаковки, указывает значения КБЖУ (калории, белки, жиры, углеводы).
7. Система проверяет корректность введённых данных.
8. Пользователь нажимает кнопку "Сохранить". 
9. Система сохраняет продукт, теперь он доступен для проверки всем пользователям.



## [ Сценарий 5 – Ведение дневника питания ]

1. Пользователь открывает карточку проверенного продукта (через сканер или историю).
2. Пользователь нажимает на иконку "Добавить в дневник питания".
3. Система открывает диалоговое окно выбора параметров приема пищи.
4. Пользователь выбирает категорию ("Завтрак", "Обед", "Ужин" или "Перекус").
5. Пользователь указывает примерный вес порции в граммах.
6. Система автоматически пересчитывает пищевую ценность для указанного веса.
7. Пользователь подтверждает сохранение записи в дневник.
8. Система обновляет общую статистику потребления за сутки.



## [ Сценарий 6 – Просмотр истории сканирований ]

1. Пользователь переходит во вкладку "История" в нижнем меню приложения.
2. Система запрашивает хронологический список последних проверок пользователя.
3. На экране отображается лента товаров с мини-статусами безопасности.
4. Пользователь прокручивает список, чтобы найти продукт, сканированный ранее.
5. Пользователь нажимает на элемент истории для повторного изучения состава.
6. Система открывает полную карточку товара со всеми характеристиками.
7. Пользователь может быстро добавить товар из истории в дневник питания.



## [ Сценарий 7 – Контроль дневной нормы калорий ]

1. Пользователь переходит во вкладку "Дневник" для анализа рациона.
2. Система отображает таблицу по всем приемам пищи за сегодня.
3. Пользователь видит шкалу прогресса, заполняющуюся по мере потребления калорий.
4. Система сравнивает текущее значение с установленным лимитом в профиле.
5. Пользователь нажимает на иконку "Статистика БЖУ" для детального анализа.
6. Система отображает соотношение белков, жиров и углеводов в виде диаграммы.
7. Пользователь принимает решение о корректировке следующего приема пищи.



## [ Сценарий 8 – Просмотр и редактирование профиля ]

1. Пользователь нажимает на иконку "Профиль" в нижней навигационной панели приложения.
2. Система отправляет запрос на сервер для получения актуальных данных аккаунта.
3. Система отображает экран профиля с именем, адресом почты и кнопками "Изменить имя" и "Сменить пароль".
4. Пользователь нажимает на кнопку "Изменить имя" и вводит новое имя в текстовое поле, после чего новое имя проверяется.
5. Пользователь выбирает функцию "Сменить пароль" и вводит старый пароль, а затем новый.
6. Система на стороне клиента проверяет длину пароля (не менее 6 символов).
7. Пользователь нажимает кнопку "Сохранить" для подтверждения изменений.
8. Клиентская часть приложения отправляет пакет обновленных данных на сервер.
9. После подтверждения изменений система возвращает пользователя на основной экран профиля.



## [ Сценарий 9 – Установка дневного лимита калорий ]

1. Пользователь открывает раздел "Профиль" и переходит в пункт меню "Настройки питания".
2. Система запрашивает с сервера текущие настройки целей и лимитов пользователя.
3. Система отображает экран настроек с текущим лимитом калорий (по умолчанию 2000 ккал).
4. Пользователь нажимает на поле ввода и указывает новое целевое значение калорий.
5. Пользователь нажимает кнопку "Сохранить" для синхронизации новых настроек с сервером.
6. Система выполняет пересчет всех текущих графиков на основе нового лимита.
7. Пользователь видит обновленную шкалу потребления, где процент заполнения соответствует новой установленной цели.
